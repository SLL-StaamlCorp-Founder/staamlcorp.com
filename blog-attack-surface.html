Beyond WebAssembly: Seven Types of Cached Executable Content at Risk

The LDB-01 vulnerability was discovered in WebAssembly and Service Workers — but the underlying vulnerability class affects seven categories of cached executable content across every major platform. Here is the full attack surface.

The Scope of the Problem

When StaamlCorp researcher Stanley Lee Linton submitted vulnerability report OE110220744757 to Apple in May 2025, the initial focus was on two content types: WebAssembly modules and Service Workers. These were the specific categories demonstrably affected in the LDB-01 (Lockdown-Bypass-01) instance within Apple's iOS Lockdown Mode.

But the underlying problem — cached executable content persisting across security policy transitions without retroactive validation — is not limited to WASM and Service Workers. The research behind U.S. Patent Application No. 19/640,793 identified seven categories of cached executable content that are structurally vulnerable to temporal security discontinuities. Each has different characteristics: different compilation behavior, different cache storage locations, different execution privileges, and different threat scenarios. All seven share the same fundamental architectural flaw: they can be cached under one security policy and executed under another.

Understanding the full scope of this attack surface is essential for any platform vendor, enterprise security team, or AI system architect who wants to understand what they are actually defending against.

The Seven Content Types

1

WebAssembly (WASM) Modules

Critical Risk

WebAssembly is a binary instruction format that runs near-native-speed compiled code in the browser. Once a WASM module is compiled from its binary representation to native machine code, the compiled artifact is cached by the browser's JIT compilation subsystem — separately from the HTTP resource cache. Under restrictive security policies like Lockdown Mode, new WASM JIT compilation is blocked. But previously compiled and cached WASM native code artifacts are not automatically invalidated. An attacker who pre-plants a malicious WASM module in the browser's compilation cache before a policy transition may retain execution capability after the transition. WASM runs in a sandboxed environment but with near-native performance and access to all APIs permitted by the current security context.

2

JavaScript Compiled Bytecode and JIT Artifacts

Critical Risk

Modern JavaScript engines (V8, JavaScriptCore, SpiderMonkey) compile JavaScript source code through multiple optimization tiers — from bytecode to optimized machine code — and cache these compiled representations. In restrictive security modes that disable JIT compilation, previously compiled JavaScript artifacts may remain in cache. The risk is compounded by JavaScript's ubiquity: virtually every web application relies on JavaScript, making the attack surface enormous. Cached JIT artifacts that predate a policy transition could execute with capabilities that would not be permitted under the new, more restrictive policy.

3

Service Workers

Critical Risk

Service Workers are JavaScript scripts registered to run in the background, independently of any web page. They intercept network requests, manage caches, and can execute even when no browser window is open for the associated origin. This persistent, page-independent execution model makes them particularly dangerous in the temporal security discontinuity context: a Service Worker registered and cached under a permissive security policy can continue to intercept network requests and execute code after the device transitions to a more restrictive policy. Service Workers have no natural expiry — they remain registered indefinitely until explicitly deregistered. The LDB-01 vulnerability directly involved Service Workers in the iOS Lockdown Mode context.

4

ML Inference Modules and Model Weights

High Risk

Machine learning inference modules — compiled representations of neural network models optimized for a specific hardware target — are increasingly cached on end-user devices for performance. These include browser-based ML inference (TensorFlow.js, ONNX Runtime Web), native on-device inference engines, and compiled model artifacts from frameworks like Core ML on Apple platforms. ML models can be weaponized: a model that was served legitimately before a policy transition could contain adversarially crafted weights designed to produce specific outputs. Under a more restrictive security policy that would have blocked the model's download or compilation, a cached model artifact presents an unvalidated execution surface. As on-device AI proliferates, this risk category will grow substantially.

5

Compiled Shaders (GPU Programs)

High Risk

Graphics shaders — programs that run on the GPU rather than the CPU — are compiled from source formats (GLSL, HLSL, WGSL) into GPU-specific binary representations and cached on disk. Shader compilation is expensive, so browsers and applications aggressively cache the compiled results. Under security policies that restrict GPU access or WebGL/WebGPU usage, newly compiled shaders would be blocked. Previously compiled and cached shader programs represent an unvalidated backdoor to the GPU execution environment. GPU-based computation has been demonstrated as a viable channel for side-channel attacks and for bypassing CPU-level security mitigations, making compiled shader persistence a non-trivial security concern.

6

Native Binary Modules

High Risk

Native binary modules — platform-compiled code distributed as shared libraries or dynamic link libraries — can be cached by package managers, runtime environments, and application frameworks. In enterprise contexts, MDM-managed applications may cache native binary components. Under security policies that restrict the execution of unsigned or third-party native code, previously cached native binary modules may not be subject to retroactive re-validation. The risk profile of native binary modules is high: they execute without the sandboxing constraints of browser-based technologies and with access to full OS APIs. A cached native module that predates a restrictive policy change represents a high-privilege execution path that bypasses the new policy's intent.

7

Microservice Container Images

High Risk

In enterprise and cloud-adjacent deployment contexts, containerized application components may be cached locally — either as full container images or as layer artifacts in container runtime caches. As enterprise security policies change (via MDM pushes, compliance requirement updates, or threat-response policy tightening), previously cached container images may contain components that would not pass validation under the new policy. This is particularly relevant in environments where edge computing or local microservice deployment is used, and where container image layer caches persist across policy transitions without automated re-evaluation.

The Six Cache Storage Locations

Understanding what content types are at risk is necessary but not sufficient. Effective remediation requires understanding where this content is stored — because each storage location has different access characteristics, different invalidation mechanics, and different interactions with security policy enforcement.

Primary Cache Storage Locations

  • HTTP Disk Cache (browser managed, origin-keyed)
  • Browser Service Worker Cache API storage
  • JIT Compilation Cache (WASM, JS bytecode)
  • IndexedDB and Origin-Private File System
  • GPU Shader Compiled Program Cache (OS-level)
  • Application-Layer Package and Module Cache

These six storage locations are managed by different subsystems: the HTTP cache is managed by the browser's networking layer, the Service Worker cache is managed by the browser's service worker API layer, the JIT compilation cache is managed by the JavaScript engine, IndexedDB is managed by the browser's storage layer, the GPU shader cache is managed by the operating system's GPU driver stack, and the application-layer package cache is managed by whichever runtime environment (Node.js, Python, etc.) is in use.

None of these subsystems communicates with the security policy management layer about policy transitions. This is the architectural root cause of the temporal security discontinuity vulnerability class. The security policy changes; the cache subsystems, operating independently, do not respond to that change.

AI and ML: The Next Frontier

The seven content types identified in the LDB-01 research represent the current state of the vulnerability class. But the most significant growth area for temporal security discontinuity risk is in AI and machine learning systems, for reasons that are distinct from the browser-layer vulnerabilities described above.

Cached Inference Modules

As AI inference moves to the edge — onto user devices, enterprise endpoints, and embedded systems — compiled inference modules become a new category of cached executable content. These modules are compiled from model architectures and optimized for specific hardware targets. Under an AI governance framework that changes (for example, in response to a new regulatory requirement or a detected model compromise), previously cached and compiled inference modules may not be subject to retroactive validation. A cached inference module compiled under a permissive governance policy could continue to execute under a more restrictive one.

Model Weights and Pipeline Components

AI model weights are not traditionally thought of as "executable content" in the same way that WASM or JavaScript are. But in the context of inference systems, model weights are operationally equivalent to executable content: they determine the behavior of the inference process, they can be adversarially manipulated, and they can be cached locally for performance. AI pipeline components — preprocessing modules, embedding models, routing logic — are similarly cached and similarly subject to temporal security discontinuity risk.

The Autonomous AI Agent Risk

The most severe risk scenario in the AI/ML domain involves autonomous AI agents operating with locally cached models and pipeline components. Unlike a web browser's Service Worker, which eventually expires or is deregistered, an autonomous AI agent's cached inference pipeline may have indefinite operational persistence — no natural cache expiry, no user interaction that triggers re-validation, and no mechanism that associates the cached components with the governance policy that was in effect when they were downloaded and compiled.

An autonomous AI agent operating in an enterprise context could, under this vulnerability class, continue to use cached inference modules and model weights from before a security policy change — executing behavior that would be blocked under the new governance framework — without any system-level mechanism detecting or preventing this.

As AI systems adopt increasingly dynamic governance frameworks — with security policies that change based on threat intelligence, regulatory requirements, and operational context — the temporal security discontinuity vulnerability class becomes not a browser-security niche problem but a fundamental AI safety concern.

Why Blanket Cache Purging Is Not the Answer

When security practitioners first encounter the temporal security discontinuity problem, the instinctive response is: just clear the cache on every security policy transition. It is a simple, obviously correct solution from a security standpoint. If there is nothing in the cache, nothing non-compliant can persist.

The problem is that blanket cache purging, while secure, is operationally catastrophic at scale. The costs are real and measurable:

Blanket Cache Purging

  • 100% cached content destroyed
  • 180+ second re-cache time for complex apps
  • Complete loss of bandwidth savings
  • User experience impact: 8.5 / 10
  • Re-fetch of all resources from network
  • Full WASM recompilation required
  • Service Worker re-registration and activation
  • GPU shader recompilation from source

Selective TSB Validation (Patented)

  • 88% cached content preserved
  • 78x faster than blanket purging
  • 92% bandwidth reduction vs. purge-and-reload
  • User experience impact: 0.3 / 10
  • Only policy-delta-affected content re-validated
  • Fast-path catches 99.97% of non-compliant items
  • Full deep validation for only 0.03% of cases
  • Compliant cached content untouched

The 8.5 out of 10 user experience impact score for blanket purging is not a theoretical concern — it represents real user behavior. In enterprise environments where Lockdown Mode or equivalent restrictive security profiles are applied and removed based on threat context (for example, enabling a restrictive profile when connected to public Wi-Fi and disabling it on trusted networks), blanket cache purging would impose this performance penalty multiple times per day for every affected user.

In practice, a security mechanism with these operational costs will not be deployed uniformly. IT administrators will create exceptions. Users will find workarounds. The security guarantee will erode precisely because the cure is too expensive to administer.

This is why the policy-delta approach at the heart of U.S. Patent Application No. 19/640,793 is architecturally significant: it achieves the same security guarantee as blanket purging — ensuring no non-compliant content executes under the new security policy — while delivering a user experience impact that is operationally sustainable.

Patent Coverage Scope

The 30 claims of U.S. App. No. 19/640,793 are written to cover the full seven-category attack surface described in this analysis. The claims are not limited to the LDB-01 instance (iOS Lockdown Mode, WebAssembly and Service Workers). The independent claims describe the general architecture: a Policy Transition Detection module, a Policy Delta Computation engine, a Temporal Security Binding framework that applies to any class of cached executable content, a Retroactive Validation Engine with a multi-stage validation pipeline, and a Mitigation and Regeneration system for disposition of non-compliant content.

The 26 dependent claims address specific embodiments: particular content types (WASM, JS, Service Workers, ML inference modules, shaders, native binaries, container images), particular policy transition scenarios (Lockdown Mode, MDM profile transitions, enterprise browser policy changes, AI governance framework updates), and particular validation mechanisms appropriate to each content type's characteristics.

Any platform vendor or enterprise security product that implements a systematic solution to the temporal security discontinuity problem — covering the seven identified content types, operating across the six cache storage locations, and handling the policy transition scenarios described in this analysis — will need to license this architecture or design around claims that comprehensively cover the solution space.
